# Architecture

## Stack

- TypeScript
- React
- Vite
- Node/Express
- PostgreSQL
- pgvector
- OpenAI-compatible APIs
- Docker Compose
- Nginx
- Smallstep CLI for local development certificates

## Docker Topology

The Docker app entrypoint is `https://localhost:8787`.

```text
browser -> nginx:443/TLS -> app-1:8787 / app-2:8787 -> postgres:5432
```

- `nginx` is the only public service.
- `nginx` load balances to `app-1` and `app-2` on the private Compose network.
- Public Docker traffic must be HTTPS only.
- App containers and PostgreSQL stay internal to Docker Compose.
- Local TLS certificates are generated by the `cert-init` Smallstep CLI service into ignored `data/tls/`.
- Docker sets `APP_INSTANCE_URLS` to probe `app-1` and `app-2`; instance health must stay free of secrets.

## Provider Configuration

- LM Studio embeddings are expected at host URL `http://127.0.0.1:1234`.
- Docker reaches LM Studio through `http://host.docker.internal:1234/v1`.
- Default embedding model: `text-embedding-nomic-embed-text-v1.5-embedding`.
- Default embedding dimensions: `768`.
- OpenAI-compatible local providers can use `not-needed` as the API key.
- If the LLM provider does not support the Responses API, set `LLM_API_STYLE=chat`.

## Data Stores

- PostgreSQL is the source of truth for users, sessions, jobs, job postings, resume versions, analysis metadata, and embeddings.
- pgvector is used for embedding storage and matching.
- `user_match_profiles` stores one searchable semantic profile per user, built from the latest resume plus application and analysis context.
- Admin semantic user profiles use `vector(768)` for `text-embedding-nomic-embed-text-v1.5-embedding`; changing embedding dimensions requires a matching schema migration and index rebuild.
- Admin user search embeds the search phrase, queries `match_user_match_profiles(...)`, and ranks matching users through a pgvector IVFFlat index while preserving exact text search as fallback.
- Analysis cache entries are stored in PostgreSQL.
- The app connects with the restricted `APP_DB_USER`; it must not connect as the Postgres superuser.

## SQL Organization

- SQL lives under `sql/`, not inline TypeScript business logic.
- Migrations live in `sql/migrations/`.
- Query files are grouped by domain:
  - `sql/jobs/`
  - `sql/job_postings/`
  - `sql/users/`
  - `sql/sessions/`
  - `sql/resume_versions/`
  - `sql/resume_chunks/`
  - `sql/user_match_profiles/`
  - `sql/analysis_cache/`
  - `sql/admin/`

## Analysis Flow

1. Extract resume text from upload.
2. Chunk resume text.
3. Create embeddings for the job query and resume chunks.
4. Store resume chunk embeddings in PostgreSQL with pgvector.
5. Rank evidence chunks using vector similarity.
6. Ask the LLM for structured HR assessment fields.
7. Compute final fit score and fit level deterministically in the server.
8. Persist job result, analysis JSON, recommendation, models, and chunk count.

## Health Flow

- `/api/health` is a lightweight public health endpoint used by app/container readiness and the login banner.
- `/api/instance-health` reports private per-instance process status for server-side probes.
- `/api/admin/system-health` aggregates storage, provider config, and all configured app instances.
- `/api/admin/system-health` requires authenticated admin access.
